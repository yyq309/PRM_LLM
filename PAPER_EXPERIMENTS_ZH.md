# 实验

我们围绕论文主线组织评测——*一个在抽象单主机 Web 模拟器中、用**零**真实目标标签、**无**泄漏、低成本学到的过程奖励/价值先验,能否通过一个受控适配器迁移出去、引导 LLM 智能体走通真实的端到端 Web 攻击链,以及这种迁移的能力与结构性边界是什么?* 据此,实验回答四个问题:**(i)** 抽象模拟器能否产出一个对"真实进展"排序正确的价值先验(§E.2);**(ii)** 该先验能否迁移到真实目标并带来逐步过程价值(§E.3);**(iii)** 它能否驱动**完整**的真实攻击链直至 root(§E.4);**(iv)** 它的结构性边界是什么(§E.5)。§E.6 证明该行为不限于单一 LLM,§E.7 诚实地报告其局限。

## E.1 实验设置

### E.1.1 第一阶段——抽象单主机 Web 模拟器

价值先验完全在一个**冻结 16 动作模式**的抽象 MDP 中训练;第一阶段不接触任何真实目标。基准为 **65 个去模板化、带家族标签的任务**,覆盖 **12 种攻击链拓扑**(难度:10 易 / 15 中 / 40 难;链深 2–12)。划分为 **45 训练 / 20 留出**,其中留出集刻意拆为 **10 个未见实例(unseen-instance)** 与 **10 个未见链(unseen-chain)**;两个未见链家族(`rce_privesc`、`leak_login`)是**训练中完全缺席的整条拓扑**,与训练集的链签名重叠为 **0**。每个任务都带 `expert_plan` 与等长的 `expert_trajectory`。

我们采用**硬模式**(紧预算 = `plan_len + 2`),使同状态下的决策**有后果**;这去除了指标饱和(例如 masked goal-rate 从饱和的 1.0 降到 0.70,oracle 价值与实际回报的相关性从 ≈0 升到 +0.38——见 §E.2)。训练流水线为 `task_config → transitions → DQN 价值 oracle → PRM 标签 → Pentest-PRM`。关键地,PRM **只**消费可观测的状态+动作特征;oracle 的 q 值**绝不**作为 PRM 输入特征。

### E.1.2 第二阶段——受控的抽象→真实适配器

第二阶段对**真实目标**运行自主回路 `propose → ψ → PRM 重排 → η → 执行 → φ`:
- **φ**:真实工具输出 → 抽象观测;
- **ψ**:自由文本的 LLM 动作 → 16 个冻结动作之一;
- **η**:抽象动作 → 具体的、按目标定制的命令;
- **安全门**(`safety.py`):对每条命令做白名单校验(限定主机/端口/传输,全程审计)。

**泄漏隔离墙**严格:PRM 与 proposer 只见可观测上下文——绝不见 oracle q 值、隐藏任务真值、walkthrough、凭据或 payload(后者只存在于 η 管线中)。

**目标。** **15 个 Docker Web 靶**(单主机单服务的真实 CVE——ThinkPHP、Struts2、Joomla、php-cgi、Drupalgeddon2 等,绑定 `127.0.0.1`)提供广度;另加 **2 台 VulnHub 虚机**(整机多步链:**DC-1**、**Toppo:1**),部署在隔离的 host-only 网段上,用于完整的 *Web 入口 → foothold → 同主机提权 → root* 链。(Raven-2 已剔除——其 CVE-2016-10033 foothold 被加固版 PHPMailer 封死;XBEN/XBOW 仅作仓库溯源,不计入论文结果。)

### E.1.3 A/B 协议与统计方法

每个真实目标结果都是两臂之间的 **A/B**,两臂共享**同一个** LLM proposer,仅在动作选择上不同:**`prm`**(PRM 对 proposer 的候选动作重排)对 **`llm_only`**(proposer 自身的排序)。指标:终局 `root_flag_captured` / goal-rate(不可作弊)、逐步进展率,以及 **top-1 排序准确率**(所选动作与 oracle 优先项一致的频率)。由于 rollout 按 episode 结构化且会分叉,我们采用 **episode 聚类置换检验**、**聚类自助 CI**,以及带 **公共随机数(CRN)** 的**重排隔离消融**——两臂在重合状态上配对。我们在 *p* 值旁同时报告**效应量**(风险差/比、比值比、Cohen's *h*),并对确证性家族施加 **Holm–Bonferroni** 校正。朴素的两比例检验会高估显著性;我们报告聚类稳健的数字。

## E.2 第一阶段:对"真实进展"排序正确的价值先验(C-A 的基础)

**RL 价值 oracle。** 在去饱和的硬基准下,oracle 的真实信号是**相对掩码内随机的 expert top-1 提升 +0.093–0.127**、**masked goal-rate 0.70**(随机 0.54)、以及**permissive(无掩码)goal-rate 0.40**(随机 0.013,约 30×)。最重要的是,**在决策真正影响结果之处**,oracle 价值与实际回报的相关性为 **Spearman +0.378**(松预算下 ≈0),且决策相关比例升至 0.49(2.4×)。oracle 是*弱、但恰在关键处真正有信息量*。

**Q\* 校验。** 对照价值迭代的**目标对齐 Q\***(只在到达目标的转移上给奖励),oracle 对进展排序正确:**top-1 一致 0.74、top-3 命中 1.00、平均 Spearman +0.45**。对*字面* Q\* 的负相关是**诱饵刷分伪影**(字面 Q\* 在终止前先刷干扰项 `path_found` 奖励),并非 oracle 的真实弱点。

**Pentest-PRM 排序质量**(强梯度提升模型,仅 oracle 标注子集)。消费 oracle 标签的 PRM 是一个强的逐步排序器:

| 划分 | 成对(Pairwise) | 排序(Rank) | 说明 |
|---|---|---|---|
| oracle_all | **0.942** | 0.799 | 结构化状态+动作特征(TF-IDF 基线仅 0.884) |
| unseen-instance | **0.980** | 0.972 | 已见链的新实例 |
| unseen-chain(难) | **0.920** | 0.617 | 训练中完全缺席的整条拓扑 |

我们**只**报告 oracle 标注子集;全集 pairwise(0.93)被夸大,因为 73% 的留出行是规则标注的 `score=0`、可被平凡地自我预测。校准后最难切片的 ECE 减半有余(unseen-chain 0.155 → 0.067)。**错误动作识别**(PRM 能否标出*不应*采取的动作?)在真正的"有效但低价值"子集上达到 **ROC-AUC 0.927 / PR-AUC 0.887**,对硬约束类错误召回完美(前置缺失 / 不安全 / 越界 / 模式缺口 / 歧义 = 1.00)。

**泛化与完整性。** 对**结构全新的链拓扑**的零样本迁移成立(**pairwise 0.858**,相对地板的排序提升 +0.32,对比同分布 0.942)——PRM 抓的是状态特征,而非链身份。多种子(3×80k)oracle 标签的排序一致性为 **0.97**。**泄漏审计**在 PRM 输入中**未发现任何隐藏真值 token**,且掩码各可观测字段时**无悬崖字段(0 cliff)**——PRM 依赖可观测上下文,而非泄漏的秘密。最后,在**闭环策略评测**中 PRM **不是**独立策略(permissive goal-rate 0.00,其余被掩码饱和):它是一个逐步**奖励模型/排序器**,其价值在于上文的排序质量——这是自主 rollout 无法测量的。

## E.3 第二阶段:抽象→真实价值迁移(C-A)

受控适配器把迁移落到实处。**模式覆盖率为 92%**,**ψ 归一化器在留出的真实 LLM 动作文本上从 49% → 78.5%**。在 15 个 Docker Web 靶上,PRM 重排 proposer 候选带来**统计显著的逐步过程提升**:汇总的、episode 聚类置换给出 **p = 0.02**(对确证性家族 Holm 校正后 **0.04**),即 PRM 逐步地优于对同一候选集的随机重排。这是 C-A 的承重证据:一个用零真实标签学到、经安全门迁移的先验,可测量地改进了真实的逐步动作选择。(我们区分*逐步*与*整局*:纯 Web 靶上的终局 goal 收益有限/常打平——明确陈述,并在 §E.4、§E.7 再议。)

## E.4 第二阶段:真实端到端攻击链(C-B,旗舰)

旗舰证据是真实虚机上的**完整**链——*Web 入口 → foothold → 同主机提权 → root*——以不可作弊的 **`root_flag_captured`** 计分。

**DC-1(自主,deepseek-chat)。** 合并两次同配置运行(n=10 + n=8 = **n=18**),**`prm` 臂在 18/18(100%)** 的 episode 中拿到 root,对 **`llm_only` 10/18(56%)**——**+44 pp** 的救场。`prm` 臂还少用约 2× 步数。关于方差的诚实说明:原始 proposer 在 DC-1 上的自主 root 率在不同运行间波动(n=10 时 40%,n=8 时 75%),而 PRM **两次都是 100%**;因此我们报告合并后的 56% 而非单次有利抽样,并把 PRM 在 DC-1 的价值定性为**可靠性**。一个**阶段拆分**解释了为何 PRM 在此有用、而在纯 Web 靶上效率中性:`prm` 臂在**两个**阶段都推进(Web ≈36% / 本地 ≈37%),但 **`llm_only` 在本地/提权阶段崩塌(9%)**,而 Web 侦察仍在做(32%)。**PRM 的全链价值正落在本地/提权阶段——恰是原始 proposer 自身排序最弱之处**,也正是纯 Web 的 A/B 无法触及的区间。

**Toppo:1 与 proposer 天花板。** 两臂自主均为 0%(proposer 始终建不起所需的 `凭据发现 → ssh` foothold),然而**确定性** proposer 在 DC-1 与 Toppo 上都拿到 root——故**适配器(φ/ψ/η)是健全的**;失败是一个 *proposer 能力天花板*(`exploit_never_proposed`),而非适配器缺陷。这干净地把"价值先验无法引导 proposer 从未提出的动作"与"迁移坏了"分开。

## E.5 结构性边界:侦察过度高估(C-C)

迁移得到的价值有一个**可刻画、且外科手术式无法修复**的局限。由于*被掩码*的抽象训练很少产生"答案已知时还侦察"的状态,PRM **高估侦察**:`web_path_enum` 的平均 PRM 标签为 **0.887**,远高于 `exploit` 的 **0.535**。在真实目标上,这表现为 PRM 添加了能干的 proposer 并不需要的探索步。我们尝试了**三种独立修复——推理期重加权、标签纠正重训、top-k 限制——三者全部失败**,无法在不破坏别处排序质量的前提下消除该偏差。多种子分析(seed 0–4)进一步显示侦察"被推进标签"的比例是**种子相关的**(均值 ≈0.25,std ≈0.06),且部署检查点的 0.609 是高位离群(种子门选择伪影),故偏差幅度本身也带噪。我们将其定性为**训练期动作*掩码*诱发的协变量漂移下的奖励模型过优化**——一个对 sim-to-real 价值迁移的可推广警示,区别于 RLHF 的策略漂移。

## E.6 跨厂商泛化(多 LLM)

为证明迁移行为**不限于单一 LLM**,我们做了一个**对称的三厂商** A/B——*同一份*当前代码、在*同样*的 7 个靶(DC-1 全链 + 6 个 Web 靶)上、llm proposer、prm 对 llm_only、零报错——覆盖 **DeepSeek-chat**(官方端点)、**Qwen-3.7-max** 与 **GPT-5.4**(后两者经 OpenAI 兼容网关)。各 headline 断言已独立对照原始日志核验。一个**proposer-conditional(随 proposer 而定)机制**解释全部三家。

**结果救场(随 proposer 而定)。** 在 **Joomla CVE-2017-8917** 上,PRM 对**全部三家**都救起了吃力 proposer 的 goal-rate:DeepSeek **1.0 对 0.4**、Qwen **1.0 对 0.4**、GPT-5.4 **0.6 对 0.0**(各 n=5;承重证据是*三家方向的一致性*,而非任一小样本 CI)。在 **DC-1** 上,救场**取决于 proposer 之弱**:DeepSeek(较弱 proposer)被救起(合并 root **100% 对 56%**),而 Qwen 与 GPT 自身已打满该靶(100% = 100%,PRM 无结果上的余量)。

**排序提升(近乎普适)。** PRM 的逐决策 **top-1 排序准确率在 21 个"厂商×靶"中 20 个上优于原始 proposer**,横跨三家。唯一例外(DeepSeek-Joomla,0.264 < 0.403)是**指标伪影**——在同一靶上 PRM 仍以 100% 对 40% 赢下真实 *goal*;top-1 是对照 oracle 启发式优先项打分的,而后者并非每个靶上的"目标真值"。

**逐步效果看靶,非一边倒。** PRM 在较长的**多步链**(DC-1、php-cgi)上改进逐步进展,但在短的**一步式** Struts2 靶上**拖累**它——那里原始 proposer 每步已直接打出唯一正确动作,PRM 的探索只会稀释进展率。三家呈现同一模式。

合起来:结果救场*取决于* proposer 弱;排序提升*近乎普适*;逐步效果随链长变化——同一个重排机制、不同的天花板,**且非 DeepSeek 特有。**

## E.7 诚实的局限:随 proposer 而定的重排价值

我们报告——但不以之为论文核心——这一局限:重排器的**整局**收益取决于 proposer 自身的排序质量。它帮到弱的/未做提示工程的 proposer(逐步 **+12 pp**,p = 0.02),但若给 proposer 明确的动作词表提示,重排器可变为**净负**(`llm_only` 66.7% 对 `prm` 39.6%,该配置下 p < 0.0001)。我们将其作为**范围局限**而非 headline,因为 (a) 它与提示工程混淆——那里的"强" proposer 是作者添加的词表提示,而非自然变量;(b) 它是已知的验证鸿沟/奖励模型过优化现象的一个实例。该效率倒置被完整报告,绝不为保护主线而隐藏。

## E.8 小结

抽象模拟器产出一个**弱但真实**的价值先验,对真实进展排序正确(对目标对齐 Q\* 的 Spearman +0.45;PRM pairwise 0.94 / unseen-chain 0.92),且**无泄漏**、**零真实标签**(§E.2)。受控适配器把它迁移到真实目标,带来**显著的逐步过程提升**(p = 0.02;§E.3),并驱动**完整的真实攻击链直至 root**,其中 PRM 的价值集中于提权阶段(DC-1 合并 100% 对 56%;§E.4)。该迁移有一个**被刻画的结构性边界**——抵抗三种修复的侦察过度高估(§E.5)——且整套行为在**三家 LLM 上复现**,由同一个 proposer-conditional 机制统辖(§E.6)。重排器的整局价值**取决于** proposer 之弱,作为局限被诚实报告(§E.7)。
