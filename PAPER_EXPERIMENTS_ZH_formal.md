# 实验

## 实验设置

本章给出评估 Pentest-PRM 的评估对象、目标靶、评估集合、模型、指标与执行约束。评估只针对一个客观判定:**在真实目标上,由抽象模拟器中无标签学得的过程奖励模型(PRM)对 LLM 智能体提议的动作做重排,是否可测量地改进其攻击过程**;"是否最终拿到 flag/root"作为下游检验报告,不作为主判据。

### 评估对象

被评估的系统是一个**重排式顾问**:LLM 智能体在每一步*提议*若干候选动作,PRM 对其打分并把最有希望者前置先试。PRM 完全在一个抽象的单主机 Web 攻击模拟器中训练——先用强化学习(DQN)训练一个**价值 oracle**,由其为候选动作的好坏打**标签**,再蒸馏出状态条件化的 PRM——全程**不接触任何真实目标的标签**,其输入**不含隐藏任务真值**(秘密路径、凭据、flag)。模拟到真实的迁移由三个适配器(φ 输出→状态、ψ 文本→16 个冻结动作、η 动作→命令)与一道安全门(`safety.py` 白名单)完成。16 动作 schema 全程冻结。

每个真实目标结果都是两个智能体的配对对决,二者用**同一个 LLM**,仅差一点:`prm`(LLM 提议、PRM 重排)与 `llm_only`(直接用 LLM 自身排序,无顾问)。任何差异只能归因于顾问本身。

### 目标靶与评估集合

目标靶分两层:**16 个真实 Docker Web 靶**(每个为一项带真实已知漏洞的 Web 服务,仅至立足点)提供广度;**3 台完整虚拟机**(DC-1、Symfonos:1、Toppo)提供深度,即*完整*的链 Web 入口 → 立足点 → 同机提权 → root。全部目标均为自有、运行在隔离 host-only 网络、经白名单校验并留痕。

在此之上划分四套评估集合,分别服务于不同研究问题:

- **D_web(主集)**:全部 16 个 Docker Web 靶,proposer 为 DeepSeek-chat,每靶 5 次运行(分母 80 次运行),用于度量过程改进的整体有效性及其逐靶分布,并以 episode 聚类统计给出池化结论。
- **D_abl(重排器隔离集)**:固定候选集、只换排序器,两种设置——*部署态*(真实 LLM proposer 给出小而有针对性的候选集,6 靶、每臂 n=30)与*压力测试*(确定性 proposer 把整个动作面一股脑喂入,11 靶、每臂 n=80)。
- **D_vm(整机集)**:3 台整机的端到端运行,DC-1 每臂 18 次、Symfonos 每臂 10 次、Toppo 以脚本式智能体确定性确认 1 次。
- **D_multi(跨厂商集)**:6 个 Web 靶 × 3 个 LLM 厂商(DeepSeek-chat、Qwen-3.7-max、GPT-5.4)的对称 A/B,用于刻画结论对模型的敏感性。

此外,Stage-1 顾问质量在模拟器留出集上单独评估(20 个留出任务,其中 10 个为训练中从未见过的攻击链形状),作为迁移的前置条件,见 RQ1。

### 模型选择

主 proposer 统一使用 **DeepSeek-chat**(temp=0.5)。本流水线对 LLM 的要求集中在动作提议与稳定的文本输出,而非多模态能力;DeepSeek-chat 单位成本低,适合 16×5 规模的重复运行。为刻画结果对模型的敏感性,D_multi 另选 **Qwen-3.7-max** 与 **GPT-5.4** 分别重跑(RQ4)。被评估的 PRM 是**部署模型** `prm_strong`,由抽象 RL 价值 oracle 蒸馏而得;落地与重排由确定性代码完成,LLM 仅负责提议。

### 评估指标

因顾问是一个**过程**奖励模型,**主要**指标是过程/阶段级的——它们度量*轨迹质量*而非仅最终是否拿旗。除特别说明,p 值为 episode 聚类置换检验,辅以集群自助置信区间与 Holm 多重比较校正。

- **逐步进展率(↑)**:真正向前推进(而非白走)的*步*占总步数的比例。
- **目标对齐进展(↑)**:仅计沿 kill-chain 前进方向的步。
- **平均到达里程碑阶段(↑)**:阶梯 0=侦察、1=漏洞、2=shell、3=命令、4=文件、5=root 上每局到达的最高阶段均值。
- **加权进展(↑)**、**立足点/shell 到达率(↑)**、**白走步率(↓)**。
- **逐决策 top-1 排序准确率(↑)**:顾问排第一的动作确为最佳(对 oracle)的比例。
- **Stage-1 成对排序准确率(↑)**:给定两动作、已知优劣,顾问排对的比例(0.5 为随机)。

**结果**指标——**整局达标率 / root 率(↑)**——含义明确、无法作弊,但作为**下游检验**报告:一个过程改进器应先被证明改进了过程。

### 执行与约束

所有真实目标运行在隔离 host-only 实验室(192.168.52.x VMnet2)与本机 127.0.0.1 Docker 上,经 `safety.py` 白名单校验并留痕;16 动作 schema 冻结;PRM 输入仅含可观测上下文,不含 oracle q 值或隐藏真值;漏洞攻略、凭据、payload 仅存在于 η 命令落地层,绝不进入 proposer / PRM / φ 上下文。每个量化结论均可追溯到 `outputs/` 下的报告。

## 实验

本章在上述四套集合上评估 Pentest-PRM,围绕四个研究问题展开:

- **RQ1(迁移有效性)**:在无真实标签训练的前提下,模拟器学得的 PRM 迁移到真实 Web 靶后,能否在多大程度上改进逐步/阶段级的攻击过程?改进在何种靶上集中?
- **RQ2(机制必要性)**:逐步增益是否来自 PRM 的*特定*排序,而非任意重排或廉价启发式?对照能否排除泄漏、提示词混淆与"顾问其实是个策略"?
- **RQ3(整机端到端与价值定位)**:迁移来的过程奖励能否驱动智能体走通*完整*的真实 kill chain 到 root?其价值集中在哪一阶段?
- **RQ4(跨厂商鲁棒性)**:上述结论是否与具体 LLM 无关,在不同厂商间成立?

随后以独立小节给出失败分析与局限。实验设置(评估集合、模型、指标、执行约束)已在前节给出,本章不再重复。所有数字均以 `outputs/` 下报告为准。

### RQ1:真实 Web 靶上的迁移有效性

**前置——适配器与顾问质量。** ψ 解释器把 LLM 自由文本映射到正确动作类型,在有标注基准上准确率 **95.5%**、在更难的留出夹具上 **78.5%**(相比未增强基线 49% 有明显提升)。Stage-1 部署顾问在留出集上的成对排序准确率为:全部留出 **0.89**(95% CI [0.84, 0.94],跨 5 个种子稳定)、训练链的新实例 **0.98**、从未见过的全新链形状 **0.80**(最弱一档,仍明显高于 0.5)。一条诚实边界:顾问是排序器而非策略,独立驱动智能体并不成功,价值在于出谋划策。

**主结果——逐步过程改进。** 在 D_web 的 16 个真实 Web 靶上,`prm` 智能体在 **52.7%** 的步上向前推进,而 `llm_only` 为 **37.6%**——领先 **+15.1 个百分点**,episode 聚类下显著:**p = 0.0012**(多重比较校正后仍显著)。逐步指标不是配角,而是主要证据:顾问在 kill-chain 阶梯上爬得更高、白走更少。表 E-1 汇总过程/阶段级指标。

**表 E-1 过程/阶段级指标(D_web,16 靶池化,prm / llm_only)。** p 为 episode 聚类置换检验,标注者方有;其余为方向性点估计。

| 过程/阶段指标 | prm | llm_only | p |
|---|---|---|---|
| 逐步进展率 ↑ | **52.7%** | 37.6% | **0.0012** |
| 目标对齐进展 ↑ | **26.3%** | 15.0% | **0.0018** |
| 平均里程碑阶段 ↑(0–5) | **1.65** | 1.06 | **0.035** |
| 加权进展 ↑ | **0.59** | 0.45 | **0.026** |

| 白走步率 ↓ | **0.35** | 0.49 | — |

八个过程指标中**四个聚类显著**(逐步、目标对齐、里程碑阶段、加权进展);立足点到达率(0.11)如实标为不显著。**结果**指标整局达标率为 prm 31% vs llm_only 12%(p = 0.005),但该增益**集中在 proposer 彻底失败的靶**(如 SSTI、Joomla),在两臂都能解的靶上则打平——这一 proposer 条件性在 RQ2、RQ3 详述。

| Struts2-S2-048 | RCE | 90 / 56 | 40 / 0 |
| php-cgi-2012-1823 | RCE | 83 / 33 | 20 / 0 |

**RQ1 takeaway:** 在无真实标签训练下,模拟器学得的 PRM 迁移到 16 个真实 Web 靶后,把逐步进展率从 37.6% 提到 52.7%(+15.1pp,聚类 p = 0.0012),八个过程指标四个聚类显著。改进**看靶**——在 proposer 彻底失败的靶上最大,在两臂都能解的靶上打平;整局达标率的增益(31% vs 12%)是 proposer 条件性的,其可信度在 RQ2 检验。

### RQ2:机制必要性——重排器隔离与对照

逐步增益可能来自*任意*重排,或被廉价脚本规则复刻。为排除这点,在 D_abl 上固定候选集、只换排序器,四个臂:原始 LLM 顺序(`llm_only`)、PRM、随机重排(`random`)、RL 价值 oracle(非 LLM 的**上界**参照)。

**表 E-3 重排器隔离:逐步进展 / goal 率(候选集固定)。**

| 排序器 | 部署态(真实 LLM proposer) | 压力测试(整个动作面) |
|---|---|---|
| `llm_only` | 0.481 / goal 0.23 | — |
| `random` | 0.500 / goal 0.33 | 0.293 / goal 0.46 |
| `oracle`(价值 oracle 上界) | 0.479\* / goal 0.27 | 0.312 / goal 0.60 |
| **`prm`(本文)** | **0.685 / goal 0.40** | 0.267 / goal 0.50 |
| *PRM vs 各臂(聚类 p)* | **+18–21 pp,p ≤ 0.007** | −2 至 −5 pp vs 随机/oracle(p < 0.01) |

\*部署态下 oracle 排的是*抽象*最优,经 ψ 映回 LLM 的具体候选文本时并不精确,故非干净上界;压力测试(它领先,0.312)才是更干净的 oracle 对比。来源 `stage2_ablation_rerank{,_llm}.json`;n = 30/臂(部署)、80/臂(压力)。

**怎么读。** **部署态**(实际运行的设置)下 PRM 的排序明显最好:0.685 vs `llm_only` 0.481(+20.4pp,p = 0.006)、`random` 0.500(+18.5pp,p = 0.007)、乃至价值 oracle 自身排序 0.479(+20.6pp,p = 0.001)。随机重排复刻不出增益,oracle 对同一候选的排序也不行——**PRM 在做真实、特定的工作**。**压力测试**(proposer 倾倒整个动作菜单)下,PRM 的侦察高估(见局限)让它把步数浪费在枚举上,被拖到随机与 oracle 之下。故 PRM 的逐步价值是 **proposer 条件性**的——当 proposer 把候选收窄到几个合理动作时(真实 LLM 正是如此)它奏效,自己给原始动作面排序时则不然。

**其他对照(表 E-4)。** 三项对照排除更平凡的解释:`llm_only` 基线即上面的对比,不再重复。

**表 E-4 其他对照(逐步效应的替代解释)。**

| 对照 | 排除什么 | 结果 |
|---|---|---|
| 无泄漏输入审计 | 顾问读到隐藏答案 | 4 176 训练 + 1 764 留出输入中 **0** 条秘密/路径/flag 泄漏;遮全部上下文仅排序 0.910 → 0.916(无降),0 个悬崖字段(`leakage_audit.json`) |
| 通用提示对照 | 成功来自 CVE 命名提示 | CVE 调教提示把 goal 率抬 **+0.375**(p = 0.002);换通用提示后提升**消失反转**(−0.094,p = 0.54,n.s.)(`improvement_proposer{,_generic}.json`) |
| 独立排序器(无 proposer) | 顾问其实是个策略 | 去脚手架后无法独自驱动智能体(goal 率 **0.000**,n = 3),收益需 proposer 预筛(`prm_policy_eval_unmasked_no_guard.json`) |

**RQ2 takeaway:** 在部署态,PRM 的排序击败原始 LLM(+20.4pp)、随机(+18.5pp)与价值 oracle 上界(+20.6pp),全部 p < 0.01——随机重排与廉价 oracle 排序都复刻不出,证明 PRM 在做真实工作。在压力测试(整个动作面)下 PRM 因侦察偏差落到随机之下,故价值是 proposer 条件性的。无泄漏、通用提示、独立排序三项对照分别排除泄漏、提示词混淆与"顾问即策略"。

### RQ3:整机端到端与价值定位

D_vm 检验迁移信号能否驱动*完整*真实 kill chain 到 root,并定位其价值。

**两台自主整机(模态各异、CI 不重叠)。** **DC-1**(自宣传 Drupal RCE → SUID `find`),合并 n=18:**prm root 18/18 = 100%** vs **llm_only 10/18 = 56%**,且约多用一倍步数。**Symfonos:1**(mail-masta LFI + SMTP 日志投毒 → RCE → SUID `/opt/statuscheck` PATH 劫持),n=10:**prm 10/10 = 100%** CI[0.72, 1.0] vs **llm_only 2/10 = 20%** CI[0.06, 0.51]。

**价值定位——阶段拆分(表 E-5)。** 两台机器上,Web 阶段两臂进展相近;**原始 LLM 在提权阶段崩塌**,而有顾问的智能体维持——同一个*过程*级特征出现在两条毫无关联的攻击链上,说明过程奖励的价值集中在最难的本地提权阶段(这是单服务靶给不出的)。

**表 E-5 跨 VM 阶段拆分:逐步进展率(prm / llm_only)。**

| VM | Web 阶段 | 提权阶段 |
|---|---|---|
| DC-1 | 36% / 32% | **37% / 9%** |
| Symfonos:1 | 51% / 48% | **100% / 24%** |

第三台机器 **Toppo**(`/admin/notes.txt` 凭据 → SSH → SUID `python`)作为第三种模态的确定性确认:脚本式智能体 100% 走通(1 次),证明适配器与顾问本身健全。

**RQ3 takeaway:** 迁移来的过程奖励驱动两台模态各异的整机自主到 root——DC-1 100%(18/18)vs 56%、Symfonos 100%(10/10)vs 20%,CI 不重叠。两台机器上 Web 阶段相近、提权阶段原始 LLM 崩塌(DC-1 9%、Symfonos 24%)而顾问维持(37%、100%),价值经验性地定位在本地提权阶段。Toppo 脚本式 100% 确认适配器健全。

### RQ4:跨厂商鲁棒性

D_multi 检验结论是否与具体 LLM 无关。我们做对称 3 厂商 A/B(DeepSeek-chat、Qwen-3.7-max、GPT-5.4),在相同搜索与重排协议下重跑。

行为在三家厂商上复现:在 proposer 彻底失败的 Joomla 上,顾问在三家全部把整局从失败救活;逐决策 top-1 排序上,**prm > llm_only 见于 21 个厂商-靶中的 20 个**;逐步进展的领先在三家上同向。一条工程注记:Qwen-3.7-max 为推理模型,需把 `max_tokens` 上限提到 16000 以避免"推理耗尽预算导致空内容"(10 次中约 3 次错误);GPT-5.4 无此问题。

**RQ4 takeaway:** 对称 3 厂商 A/B(DeepSeek/Qwen-3.7-max/GPT-5.4)下,顾问行为复现——Joomla 三家全部救活,top-1 prm > llm_only 见于 20/21 厂商-靶,逐步领先同向。效应不系于单一 LLM。

### 失败分析与局限

**逐靶失败。** 顾问的增益**看靶**:在 Drupalgeddon2(逐步 41 vs 100)、Tomcat-12615/Gitea/WebLogic 上 `llm_only` 反而占优;ThinkPHP-5 上原始 LLM 连*目标*都赢(80% vs 20%)。这些既不藏也不平均——逐靶 n=5 功效不足,只看方向,池化才是确证。

**局限 L1——侦察高估(诊断,范围受限)。** 迁移来的价值**高估侦察**:`web_path_enum` 平均标签 **0.89** ≫ `exploit` **0.54** ≫ `command_execution` / `privilege_escalation` **0.04**,因为遮蔽式抽象训练很少产生"已知却仍侦察"的状态。这是 RQ2 压力测试中 PRM 在原始动作面劣于随机的成因。我们将其刻画为**训练期动作遮蔽诱发的协变量偏移下的奖励模型过优化**,并提供可复现诊断。**诚实范围:** 我们证明三种*事后*修复(推理降权 / 标签纠正重训 / top-k 限制)均**失败**(标签纠正重训后 `web_path_enum` 0.59 仍远高于决定性动作,且偏差随种子而变,均值 0.54、标准差 0.11),但**不**声称该失效不可由替代训练/模拟器设计避免,也**不**声称其他 LLM-pentest 系统必然撞上。故它是*本迁移配方*的受限局限,作为诚实告诫报告,**非**第三创新点。

**局限 L2——重排器的结果增益是 proposer 条件性的。** 顾问对*最终结果*的好处取决于 LLM 自身排序本来有多好:它明显帮到弱/未调教的 LLM(逐步 +15pp);但一旦用动作词表提示*调教* LLM,proposer 自己就变好(goal 率 0.16 → 0.53、白走步率 0.52 → 0.32),而在只考排序的隔离测试里 PRM 逐步进展(0.27)并不优于随机重排(0.29)。我们完整报告这一倒置(它(a)与提示工程混淆,(b)是验证器/过程奖励文献中已知现象的具体案例),而**不**以之为论文核心。

**失败分析 takeaway:** 顾问增益看靶,在若干 RCE 靶上 `llm_only` 反占优,如实计入。两条诚实局限——侦察高估(L1,事后三修复均失败,但未证其不可避免)、重排器结果增益的 proposer 条件性(L2,与提示工程混淆)——作为诊断与告诫报告,非创新点。
