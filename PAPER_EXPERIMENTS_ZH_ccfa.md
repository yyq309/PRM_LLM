# 实验

## E.1 概述

本文研究一个自主攻击单主机 Web 服务器的智能体:发现漏洞、取得立足点(一个 shell),并在同机提权到 *root*。大语言模型(LLM)在每一步可*提议*若干候选动作,但其排序常不理想,产生白走步数。我们提出一个低成本**顾问**,对 LLM 提议的候选做**重排**,把最有希望者前置先试。顾问完全在一个抽象模拟器中训练,不使用任何真实目标的标签,其输入不含隐藏任务真值。本章检验这个在模拟器中学得的顾问能否在真实机器上给出有用建议。

我们将该顾问称作 **PRM**(过程奖励模型),它对每一*步*的好坏打分,而非只看最终结果。一个限定贯穿全章:顾问是一个 **proposer 条件性**的重排器。当一个真实 LLM proposer 已把候选收窄到少数合理动作时,它改进逐步质量;当它必须为完整动作面排序时则不然(E.4 压力测试)。其迁移是一个救起弱 proposer 的重排器的迁移,而非一个通用独立排序器的迁移。

实验围绕三个**研究问题(RQ)**展开,各对应一个创新点;E.7 做跨厂商鲁棒性的压力测试,E.6 给出失败分析与局限。

| RQ | 问题 | 节 | 创新点 |
|---|---|---|---|
| **RQ1** | 只在抽象模拟器中训练的过程评估器,能否捕捉真实的步骤质量且无泄漏? | E.3 | C1 |
| **RQ2** | 这个无标签评估器能否迁移到真实目标,并可测量地改进逐步/阶段级过程? | E.4 | C1、C2 |
| **RQ3** | 迁移来的过程奖励能否驱动智能体走通*完整*的真实 kill chain 到 root,其价值集中在哪? | E.5 | C3 |

## E.2 实验设置

**模拟器(顾问的训练环境)。** 我们不在标签稀缺的真实靶上训练,而为单主机 Web 攻击构建抽象模型:一个简化世界,配一份**固定的 16 种动作菜单**(枚举路径、验证漏洞、执行命令、提权等)。我们生成 **65 个攻击任务**,覆盖 **12 种攻击链形状**,留出 20 个任务做测试,其中 10 个使用顾问在训练中从未见过的攻击链形状,以测试泛化而非记忆。模拟器中依次训练:一个强化学习(DQN)**价值 oracle**(学习任意状态离目标多近),由它为每个候选动作的好坏打**标签**,再蒸馏出状态条件化的 **PRM**。顾问全程只接受可观测信息,不接触 oracle 的内部 q 值或隐藏答案;16 动作 schema 全程冻结。

**适配器(模拟器建议如何到达真实目标)。** 三个适配器在模拟器与真实机器之间搭桥,外加一道安全门:**读取器** φ 把真实工具的原始输出转成模拟器状态格式;**解释器** ψ 把 LLM 自由文本动作映射到 16 种固定动作之一;**执行器** η 把选定动作类型变成针对该具体目标的真实命令;**安全门**对每条命令做白名单校验,确保智能体只触碰隔离实验室机器,并留痕。漏洞攻略、凭据与 payload 仅存在于 η 命令落地层,不进入 proposer、PRM 或 φ 上下文。

**目标靶。** 目标分两层:**16 个真实 Docker Web 靶**(每个为一项带真实已知漏洞的 Web 服务,仅至立足点)提供产品广度;**3 台完整虚拟机**(DC-1、Symfonos:1、Toppo)提供深度,即*完整*的链 Web 入口 → 立足点 → 同机提权 → root,这是单服务靶给不出的(表 1)。

**衡量方式与公平对比。** 每个真实目标结果都是两个智能体的配对对决,二者用**同一个 LLM**,仅差一点:`prm`(LLM 提议、顾问重排)与 `llm_only`(直接用 LLM 自身排序,无顾问)。任何差异归因于顾问本身。`llm_only` 是主基线;在能锐化结论处再加两个参照:**随机重排**对照(把同样候选随机打乱,分离顾问的*特定*排序起作用还是仅重排即有用)与 RL **价值 oracle**(同一候选上的非 LLM 排序参照,即 PRM 蒸馏自的老师)。该 A/B 度量 proposer 与 PRM 的*联合*效应:两臂共享同一个 proposer,故它度量端到端的部署增益,而非把迁移效率从 proposer 的补偿中分离出来。分离迁移效率需要一个 oracle 训练或零标签基线 PRM 作为对照,本文未运行,故迁移*效率*仍为开放问题,端到端部署*增益*则已确立。

因顾问是过程奖励模型,主要指标为过程/阶段级,度量轨迹质量而非仅最终是否拿旗:逐步进展率(↑,真正前进而非白走的步占比)、到达阶段 / shell 到达率(↑,在 kill-chain 阶梯侦察→漏洞→shell→命令→文件→root 上爬多高、多常到达立足点)、逐决策 top-1 排序准确率(↑,顾问排第一的动作确为最佳的比例,对 oracle)、白走步率(↓)。结果指标整局达标率 / root 率(↑)含义明确、无法作弊,作为下游检验报告;一个过程改进器应先被证明改进了过程。

**统计方法。** 一次运行是一串相关联的步骤而非独立抛硬币,故聚类单元定义为**每靶一个集群**:D_web 为 16 靶 × 5 次共 16 个集群、80 次运行/臂,D_abl 压力测试为 11 靶 × 80 次/臂。p 值由 **episode 聚类置换检验**给出:在保持靶内配对结构的前提下,对集群标签做置换重采样,辅以集群自助置信区间,并以**公共随机数**把两臂配对在相同处境上。多重比较以 Holm 校正,假设数 m 按每节实际检验数计;RQ2 主结果 +15.1pp 在 Holm 校正后仍显著(校正后 p = 0.0048,m = 4)。全章报告这些保守数字,每个量化结论可追溯到 `outputs/` 下的报告。

## E.3 RQ1——模拟器中学得的过程评估器的忠实性与无泄漏性

模拟器学得的顾问获得了一种真实(虽不强)的"哪个动作能推进"的排序能力,且全程无泄漏。三方面证据如下。

**(a) 它对"进展"排序正确。** 将价值 oracle 与模拟器的数学最优解(价值迭代算出)对比:它的单个最佳猜测只有 32% 与最优一致,但最优动作有 94% 落在它的前 3 名内,整体排序与最优的排序相关性为 +0.46(0 随机,1 完美)。这是一个粗糙但真实的进展信号,可靠地把好动作留在靠前位置,而这正是一个重排器所需:它只需缩短候选清单,自己不必是 oracle。

**(b) 顾问本身是个好排序器。** 以成对准确率衡量(给定两动作、已知优劣,顾问排对的比例,0.5 随机、1.0 完美):部署的顾问在全部留出任务上达 0.89(95% CI [0.84, 0.94],跨 5 个训练种子稳定),在训练链的新实例上 0.98,在从未见过的全新链形状上 0.80(最弱一档,仍明显高于 0.5)。一个偏好损失训练变体能把"新链"这档提到 0.93,说明尚有未部署的提升空间。全章报告部署模型的数字,因下面每个真实目标结果都由它驱动。顾问是排序器而非策略:让它完全独立驱动智能体,它作为独立策略并不成功,价值在于出谋划策。

**(c) 无泄漏。** 审计确认顾问输入不含任何隐藏答案(无秘密路径、凭据或 flag),且遮掉任意单个可观测字段只让它平缓退化。其能力来自可观测上下文。

**表 2 Stage-1 顾问质量(部署模型)。** 留出集评估;PRM 成对准确率为 5 个种子上的结果。

| 检查项 | 数值 | 含义 |
|---|---|---|
| Oracle top-1 vs 最优 | 0.32 | 精准命中最佳动作只是中等 |
| Oracle top-3 vs 最优 | 0.94 | 最优动作几乎总在靠前位置 |

## E.4 RQ2——向真实目标的迁移与逐步过程改进

一个无标签、模拟器训练的重排器**以 proposer 条件性顾问的身份**迁移到真实目标:当真实 LLM proposer 已把候选收窄时,它把逐步进展从 37.6% 提到 52.7%(+15.1pp,D_web,16 靶,每靶 5 次,共 80 次运行,episode 聚类 p = 0.0012,Holm 校正后 p = 0.0048 仍显著);而当强制它为完整动作面排序时(本节末的压力测试),它落到随机之下。本节当场刻画这一 proposer 条件性,而非留到后文。

迁移的前提是适配器可用:解释器 ψ 把 LLM 自由文本映射到正确动作类型,在有标注基准上准确率 95.5%、在更难的留出夹具上 78.5%(相比未增强基线 49% 有明显提升)。表 2a 汇总过程/阶段级指标;最终结果率是下游检验。

**重排器隔离消融(D_abl)。** 逐步增益原则上可能来自*任意*重排。为排除这点,固定候选集、只换排序器,四个臂:原始 LLM 顺序、PRM、随机重排、RL 价值 oracle。两种设置:*部署态*(真实 LLM proposer 给出小而有针对性的候选)与*压力测试*(确定性 proposer 把整个动作面一股脑喂入)。

†部署态的 `oracle` 排的是*抽象*最优,经 ψ 映回 LLM 的具体候选文本时并不精确,故非干净上界;PRM 在部署态对其领先 +20.6pp(p = 0.001)仅作参考,部署态的两个干净对比是 vs `llm_only` 与 vs `random`。

*图 8。重排器隔离消融——只换排序器,候选集固定。左(部署态,真实 LLM proposer):PRM 在 vs `llm_only` 与 vs `random` 两个干净对比上领先,p < 0.01。右(压力测试,整个动作面):PRM 因侦察偏差落到随机与 oracle 之下。*

在 proposer-失败靶上(如 Flask-SSTI 的 `llm_only` 逐步 0%),没有"好候选"可排;顾问重排的仍是同一套固定候选集机制——proposer 即便整体弱,在该步仍会提出若干候选(含正确动作),顾问把其中的正确动作前置,而非回落到更大的动作空间。表 2d 的 6 个部署靶与 SSTI/Joomla 的救场使用同一套固定候选集机制。

**其他对照。** 三项对照排除逐步效应更平凡的解释(`llm_only` 基线即上面的对比,不再重复)。

**表 2c 其他对照(逐步效应的替代解释)。**

| 对照 | 排除什么 | 结果 |
|---|---|---|
| 无泄漏输入审计 | 顾问读到隐藏答案 | 在 4 176 训练 + 1 764 留出 PRM 输入中 0 条秘密/路径/flag 泄漏;遮掉*全部*上下文几乎不动(排序 0.910 → 0.916,无下降;诊断 0.907 → 0.886;0 个悬崖字段)。来源 `leakage_audit.json` |
| 通用提示对照(proposer 退化) | 改进来自提示词工程而非排序 | CVE 调教提示把 proposer 的 goal 率抬 +0.375(p = 0.002)、白走 −0.194(p = 0.012);换无内容通用提示后增益消失(goal −0.094,p = 0.54,n.s.;白走 +0.027,p = 0.74,n.s.)。来源 `stage2_improvement_proposer{,_generic}.json` |
| 独立排序器(无 proposer) | 顾问其实是个策略 | 去脚手架后无法独自驱动智能体(goal 率 0.000,n = 3,初步);其收益需要会预筛的 proposer(表 2d)。来源 `prm_policy_eval_unmasked_no_guard.json` |

无泄漏输入审计是本文最强的反泄漏证据:5940 条 PRM 输入中 0 条隐藏真值泄漏,且遮掉全部上下文后排序准确率不降反微升(0.910 → 0.916),说明顾问的排序能力不依赖任何隐藏字段。通用提示对照测的是 proposer 退化:CVE 命名提示提升的是 proposer 本身的 goal 率,换通用提示后提升消失,印证结果增益的 proposer 条件性;它不直接排除 CVE 命名泄漏,反泄漏由输入审计承担。独立排序器一行 n=3,仅作初步排除"顾问即策略",更大 n 会进一步收紧该结论。

迁移效率(把迁移与 proposer 补偿分离)需要 oracle 训练或零标签基线 PRM 作对照,本文未运行,故迁移效率仍开放;端到端部署增益(+15.1pp 逐步、部署态 +18–20pp)已确立。本文未测量标签纠正等修复是否改变了部署态 0.685 或压力测试 0.267(见 §E.6)。

**局限 L2——结果增益的 proposer 条件性。** 顾问对最终结果的好处取决于 proposer 自身排序本来有多好。在只考排序的隔离测试(表 2d 压力列)里,PRM 逐步进展 0.267 与随机 0.293 相当;用动作词表提示调教 proposer 后,proposer 自身即变好(goal 率 0.16 → 0.53,白走步率 0.52 → 0.32),顾问的结果增益随之收缩。结果层面的帮助与提示工程混淆,且仅在 proposer 弱时显著;逐步层面的迁移信号(E.4 主结果)不依赖于此。

## E.5 RQ3——整机端到端到 root 与价值定位

迁移来的过程奖励驱动两台模态各异的整机自主到 root,其帮助集中在最难的提权阶段。

**端到端成功(D_vm)。** 在 **DC-1**(自宣传 Drupal RCE → SUID `find`),合并 18 次:prm root 18/18 = 100%(Wilson CI [0.82, 1.0])vs llm_only 10/18 = 56%(CI [0.34, 0.75]),Fisher 精确检验 p = 0.0034,且约多用一倍步数。LLM 单独 root 率在不同批次波动(一批 40%、另一批 75%),而有顾问的智能体两批都 100%,故报告合并 56%。在 **Symfonos:1**(mail-masta LFI + SMTP 日志投毒 → RCE → SUID `/opt/statuscheck` PATH 劫持),10 次:prm 10/10 = 100%(CI [0.72, 1.0])vs llm_only 2/10 = 20%(CI [0.06, 0.51]),Fisher p = 0.0007。两台机器的 prm 与 llm CI 均不重叠。Symfonos 的 prm CI [0.72, 1.0] 在 n=10 时偏宽,一次失败即把下界拉到约 0.65;两台机器的 100% 点估计建立在小样本上,且共用同一套适配器,可能存在共享失效模式,因此 100% 应读作"在本规模下未观测到失败"而非渐近真值。来源 `stage2_fullchain_dc1{,_n10}.json`、`stage2_fullchain_symfonos{,_b}.json`。

**价值定位——阶段拆分。** 把逐步进展按阶段拆开(表 3a):两台机器上 Web 阶段两臂进展相近,而原始 LLM 在提权阶段崩塌、有顾问的智能体维持。这与"过程奖励的价值集中在本地提权阶段"一致;但该定位基于 n=2 台机器,且二者共用同一套适配器,尚不足以排除单台机器或适配器特有效应。

**表 3a 跨 VM 阶段拆分:逐步进展率(prm / llm_only)。**

| VM | Web 阶段 | 提权阶段 |
|---|---|---|
| DC-1 | 36% / 32% | 37% / 9% |
| Symfonos:1 | 51% / 48% | 100% / 24% |

![图 7](figures/fig7_phase_split_2vm.png)

*图 7。两台自主 VM 上按阶段的逐步进展。两臂在 Web 阶段进展相近;原始 LLM 在提权阶段崩塌(DC-1 9%、Symfonos 24%),而有顾问的智能体维持(37%、100%)——两条不相关的攻击链上同一个过程级特征。*

## E.6 局限 L1——迁移来的评估器高估侦察

顾问系统性地高估侦察:`web_path_enum` 平均标签 0.89 ≫ `exploit` 0.54 ≫ `command_execution` / `privilege_escalation` 0.04(图 4)。遮蔽式抽象训练很少产生"已知却仍侦察"的状态,顾问因而学成"侦察几乎总是有价值"。这是 RQ2 压力测试中 PRM 在完整动作面劣于随机的成因。诊断假设是编码器把"已发现路径数"等侦察特征当作价值代理并向上外推;这一解读与训练期动作遮蔽诱发的协变量偏移下的奖励模型过优化一致 [gao2023scaling],但本文不将其作为已定论的因果机制。

![图 4](figures/fig4_recon_overvaluation.png)

*图 4。顾问对各动作类型的平均打分。侦察/踩点类动作(红)被打得远高于决定性后段动作——执行命令(0.04)、提权(0.04),这正是我们追溯到模拟器训练分布的侦察高估。*

三种事后修复——推理时给侦察降权、标签纠正重训、在有更好动作时禁止侦察(top-k 限制)——均未消除该偏差。标签纠正重训后 `web_path_enum` 仍为 0.59,远高于决定性动作的 0.04;多种子检查显示偏差大小本身不稳定(均值 0.54,标准差 0.11,n=5 种子),0.54 显著高于 0.04 的决定性动作水平,但 n=5 限制了普适性。三种修复都作用于标签或推理而未触及学到的表征,其失败与"偏差栖身于学得的表征而非标签"一致。本文未测量标签纠正是否改变了 §E.4 的部署态 0.685 或压力测试 0.267,该端到端影响未测量。该偏差是否为无标签抽象迁移的内在属性、还是可被替代训练/模拟器设计规避,留作开放问题(一个受控的替代设计对照实验可定论)。

## E.7 RQ4——跨 LLM 厂商鲁棒性

在对称 3 厂商 A/B(D_multi:DeepSeek-chat、Qwen-3.7-max、GPT-5.4,7 个 Web 靶,相同搜索与重排协议)下,顾问行为复现,统于一条规则:救起弱 proposer、对强 proposer 多余。

在 proposer 彻底失败的 **Joomla** 上,顾问把三家的整局从失败救活(DeepSeek、Qwen 从 0.40 到 1.00,GPT-5.4 从 0.00 到 0.60,表 4)。在 **DC-1** 上,较弱的 DeepSeek 被救起(100% vs 56%),而 Qwen、GPT 自身已饱和(100%),顾问无结果余量可补(表 5)。逐决策 top-1 排序上,prm > llm_only 见于 21 个厂商-靶对(3 厂商 × 7 靶)中的 20 个;唯一反例是 DeepSeek-Joomla(该靶上有顾问的智能体在真实目标上仍以 100% vs 40% 胜出,系排序指标的测量伪影)。20/21 的二项检验 p = 1.1×10⁻⁵;但这些对并非独立(共享同一批靶、LLM 误差相关),故该 p 高估了精度,应读作方向一致性而非严格独立检验。

## E.8 小结

低成本的抽象模拟器产出一个真实、无泄漏的过程评估器(E.3);它以一个 proposer 条件性的过程奖励重排器的身份**迁移**到真实目标,在真实 LLM proposer 已收窄候选时把逐步进展从 37.6% 提到 52.7%(+15.1pp,Holm 校正后仍显著),救起弱 proposer,在 proposer-强靶上则反转(E.4);它**驱动两台模态各异的整机自主到 root**(DC-1 100% vs 56%、Symfonos 100% vs 20%),价值与最难的提权阶段一致(E.5);它有一个被刻画清楚的失灵模式——高估侦察,抵抗三种事后修复(E.6);整幅图景在三个不同 LLM 上复现(E.7)。迁移的是一个救起弱 proposer 的重排器,而非一个通用独立排序器;把迁移效率从 proposer 补偿中分离,留作开放问题。
