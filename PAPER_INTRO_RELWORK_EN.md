## Introduction

Automated penetration testing aims to reproduce, end to end, what a human red-teamer does against a target: enumerate exposed surface, locate an exploitable weakness, gain a foothold, escalate privileges, and reach root. Large language models have made this goal newly tractable. By turning shell transcripts, scanner output, and tool documentation into a planning context, an LLM agent can reason about the next action in free text and drive a multi-step intrusion with little hand-coded logic. Systems built on this template chain reconnaissance, exploitation, and post-exploitation across realistic and CTF-style hosts [pentestgpt, hacksynth, rapidpen], and a growing body of benchmarks measures how far such agents get [autopenbench, pentesteval]. The promise is broad generalization across products and CVEs, though in practice current systems still need per-target plumbing to fire a given exploit.

The difficulty is that a penetration test is a long-horizon decision process under a sparse signal. Success — a shell, a flag, root — arrives only at the end of a chain that may span dozens of actions, while most of those actions are silently wrong: redundant scans, exploits aimed at the wrong service, post-exploitation moves that the current foothold cannot support. An agent graded only on the terminal outcome receives no signal about which intermediate step was the mistake, and recent failure-mode analysis traces a large share of agent breakdowns to exactly these per-step errors accumulating over a long context [excalibur]. The response in the literature has been to add structure around the LLM: task trees and reasoning modules that counter state loss [pentestgpt], planner/summarizer loops [hacksynth], an external classical planner that constrains long-horizon paths [checkmate], evidence-guided attack-tree search [excalibur], and handoff protocols that compress prior knowledge for a fresh agent instance [chap]. These improve how the agent organizes its trajectory, but the agent still chooses its next move without a learned estimate of how much that move advances the kill chain.

Step-level evaluation of reasoning trajectories is the problem that process reward models (PRMs) were introduced to solve. In mathematical and tool-use reasoning, scoring each intermediate step rather than only the final answer improves both verification and policy learning [lightman, letsreward, prmsurvey], and the idea has been carried into interactive agents [agentprm, steca, qvalue]. The obstacle to applying it in penetration testing is supervision. Process labels in prior work come from dense human annotation [lightman] or from Monte-Carlo and MCTS rollouts that estimate a step's value by sampling continuations [agentprm, qvalue], and both are impractical for offensive security: every rollout is a live exploitation attempt against a real host, so per-step ground truth has no cheap real-target oracle. The strongest learning-based pentest agents therefore optimize against outcome or sparse rewards. Pentest-R1 is the closest such agent, training a reasoning policy with two-stage offline-then-online RL on 500+ walkthroughs and CTF tasks but with no process reward model; its reward credits the whole trajectory rather than the individual decision, so it cannot localize which of dozens of steps was the mistake, the error class the failure-mode work attributes breakdowns to [pentestr1, excalibur]. Open benchmarks likewise score outcomes or per-stage milestones rather than per decision [autopenbench, pentesteval]. A separate body of work applies reinforcement learning directly, learning value-based or hierarchical attack policies in network simulators and emulators [nasimemu, hierdrl, hierexpert, dqn] and even guiding post-exploitation on real environments [raiju]; these transfer a learned attack policy to act, whereas what a process-reward setting needs is a learned per-step evaluator that can grade the moves a real LLM agent proposes against a real target.

Placing these lines on two axes — whether the per-step signal is a learned value, and whether the domain is offensive penetration testing — leaves one cell unoccupied. Prompted CTF judges score offensive moves but learn nothing; trained PRMs exist for math, tool-use, and generic agents [agentprm, steca, qvalue] but not for pentest; planning and harness methods order candidate actions with a hand-specified search heuristic or symbolic planner over a structured action set, not a learned value over the free-text moves an LLM actually proposes. We occupy this cell with a proposer-conditional reranker: a simulator-trained PRM that re-ranks a real LLM agent's already-narrowed candidates through a full real kill chain. We treat the per-step value learned by an RL agent in an abstract simulator as a source of process labels that requires no real-target annotation, train a state-conditioned PRM on those labels, and transfer it to real targets to re-rank the actions a real LLM proposes, acquiring step-level supervision without a single real-target label and without leaking hidden task state into the evaluator.

Our contributions are as follows.

- We lift autonomous pentest from a terminal success signal to a dense, per-step process reward: a model (the Pentest-PRM) that scores how much each candidate next-action advances the kill chain and re-ranks the agent's move accordingly. To our knowledge this is the first trained, per-step process-reward model for offensive penetration testing, distinct from prompted CTF judges, from trained defensive-code PRMs, and from process-reward models built for math, tool-use, or generic agents [agentprm, steca, qvalue]. On the candidate set a real LLM proposer has already narrowed, the advised agent makes forward progress on 52.7% of its steps versus 37.6% for the unaided LLM (+15.1pp, episode-clustered p = 0.0012); on the full action surface the same model does not beat a random re-ordering, a proposer-conditional scope we characterize in §5. This is a per-step process gain, not an outcome improvement on these single-service boxes.

- We acquire the reward without real-target labels. We train a DQN value oracle in an abstract single-host-web MDP, read its per-step values as candidate-action quality labels, train a state-conditioned PRM on them, and transfer it to real targets through a gated φ/ψ/η adapter (φ: real output → abstract state; ψ: LLM text → 16 frozen action types; η: action → concrete command). The pipeline produces a working process reward at zero real-target annotation cost (it does require a full RL value-oracle run and a seed-gate selection) and with no leakage of hidden task ground truth into the evaluator's input. The ψ interpreter reaches 95.5% on a labeled benchmark and 78.5% on harder held-out fixtures, and a leakage audit confirms the re-ranking input carries no secret state and degrades gracefully under masking.

- We validate the transferred reward on whole machines. Re-ranking a real LLM agent's moves, it lets the agent complete real kill chains to root on whole-machine VulnHub VMs of distinct attack modality where the unaided agent stalls (DC-1: 18/18 root vs 10/18; Symfonos:1: 10/10 vs 2/10), and a per-phase split localizes its value to the same-host privilege-escalation phase, where the unaided LLM reaches the foothold but stalls. We report two VMs; a third, Toppo, exposes a proposer ceiling the reranker cannot lift (0% on both arms, because the exploit is never proposed). The per-step ranking lift reproduces across three LLM vendors (top-1 prm > llm-only on 20/21 vendor-boxes); outcome rescue appears wherever the proposer has headroom (e.g. Joomla, all three vendors).

These results carry two limitations we surface here and bite to the strongest claims in §5–§6. The PRM is a proposer-conditional reranker, not a standalone ranker: fed the raw action surface it falls below a random re-ordering, and its per-step gain does not, on these single-service boxes, convert into more episodes reaching the full goal. Its value labels also inherit an optimistic valuation of reconnaissance from the simulator reward, harmless in the abstract MDP but useless on a real target; §6 diagnoses this as a seed-dependent sim-to-real reward-design gap, and the phase-split attributes the key whole-machine gain to privilege escalation rather than recon.

Section 5 reports the Stage-1 evaluation, real-target transfer, whole-machine validation, cross-vendor robustness, and the recon over-valuation limitation. ## Related Work

### A. LLM-driven penetration testing

LLM agents reframe penetration testing as free-text reasoning over tool output, and the central engineering problem is keeping a coherent plan over a long, noisy trajectory. The work closest to our novelty is the learned-reward line. Pentest-R1 trains a reasoning policy with two-stage RL, offline imitation on 500+ walkthroughs followed by online RL on CTF tasks [pentestr1]; its reward is computed at the trajectory level over success and milestone outcomes, so it cannot attribute a failure to the specific decision that caused it, which the failure-mode analysis identifies as a dominant breakdown mode [excalibur]. We add the component this leaves out, a learned per-step value over the candidate actions, and recover it without the online real-host rollouts an outcome RL loop would require on a live target.

A second line offloads global structure to an external reasoner and, in doing so, imposes a per-step ordering of candidate actions — but a hand-specified one. CHECKMATE drives the agent with a classical planner that constrains long-horizon paths [checkmate], and Excalibur, building on its failure-mode study, adds a tool-skill layer, task-difficulty assessment, and evidence-guided attack-tree search [excalibur]. Their ordering is a search heuristic or symbolic planner over a structured action set, not a learned value over the free-text moves an LLM actually proposes, which is the signal we supply. The remaining systems target trajectory coherence rather than per-step scoring: PentestGPT introduces a task-tree and reasoning modules to counter state loss over long flows [pentestgpt], HackSynth pairs a Planner with a Summarizer in a dual-module loop [hacksynth], RapidPen starts from a single IP and combines ReAct planning, retrieval over success cases, and a command/execution feedback loop to obtain a shell [rapidpen], and CHAP compresses prior knowledge into a handoff protocol for a fresh agent instance to limit context bloat [chap]. Progress is tracked by benchmarks that grade outcomes or per-stage milestones rather than individual decisions: AutoPenBench over 33 vulnerability tasks [autopenbench] and PentestEval over 346 tasks with stage-level scoring of information gathering, weakness screening, attack decision, and exploit generation [pentesteval]. Our contribution is orthogonal to these: a learned per-step process reward that re-ranks the proposer's move, rather than another planning, harness, or benchmarking layer.

### B. Process reward models

Process reward models score the quality of each intermediate step instead of only the final answer. Lightman et al. show that step-level human process supervision outperforms outcome-only reward for math reasoning, which is also why a PRM supports test-time step selection and re-ranking, not only training [lightman]; a parallel line couples a step-level reward model with heuristic greedy search over reasoning paths [letsreward], and a recent survey organizes the design space of data generation, PRM architecture, and test-time and RL usage [prmsurvey]. Because human step labels are expensive, ThinkPRM makes the verifier generative, producing a long chain-of-thought verification from few process labels [thinkprm], while several methods estimate process value by sampling: AgentPRM uses Monte-Carlo rollouts to label process reward for interactive agents and optimizes the policy in an actor-critic/RLHF style [agentprm], STeCa compares step-level rewards to find bad steps and reflects to build calibrated trajectories [steca], and step-level Q-value models annotate steps via MCTS and train a DPO ranker over actions [qvalue]. Across these methods the labels come either from human annotation or from MC/MCTS rollouts of the very task being scored, neither of which carries over to offensive security, where each rollout is a live exploitation attempt and per-step ground truth has no cheap oracle on a real host. We instead read process labels from the value function of an RL agent trained in an abstract simulator. In the security setting the nearest reward-model work lies on either side of the cell we target: prompted judges score offensive moves but learn nothing, and trained PRMs to date address defensive code rather than offensive action, so none is a trained, transferable, label-free per-step process reward for offensive pentest.

### C. Reinforcement learning for automated penetration testing

A separate line trains RL agents to act as attackers. Value-based deep RL provides the backbone [dqn], and NASimEmu trains generalizable network-attack agents under a simulation-to-emulation setting, transferring a learned policy from the simulator to an emulated network [nasimemu]. To cope with the large action space and long horizon, hierarchical agents decompose the policy into high- and low-level controllers for faster convergence than flat DQN [hierdrl], and adding expert priors as rules and a knowledge graph further constrains and suggests actions in a two-layer agent [hierexpert]. Raiju moves beyond simulation, using A2C/PPO to select actions and invoke Metasploit modules for privilege escalation, hash dumping, and lateral movement on real environments [raiju]. In each case the trained artifact is an attack policy that is deployed to act. Our use of RL differs in role: the value oracle is a label source, not the deployed agent. We read its per-step values as process-reward targets, train a state-conditioned PRM on them, and transfer that evaluator, rather than the RL policy, to grade and re-rank the moves a real LLM agent makes against real targets.
